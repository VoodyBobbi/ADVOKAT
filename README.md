# Агент-Адвокат

Telegram-бот, который даёт оценку содеянному по закону — коротко, по-человечески, минуя горы юридической волокиты. Описываешь ситуацию своими словами — бот задаёт пару уточняющих вопросов и даёт квалификацию: какие статьи применимы, какие есть нюансы, какова перспектива и насколько он уверен в выводе.

Бот не выносит вердиктов и не утверждает, что человек «совершил преступление» — он только квалифицирует описанные действия, опираясь на текст закона и презумпцию невиновности: всё, что не было прямо сказано, считается отсутствующим.

## Как это работает

Бот использует RAG (Retrieval-Augmented Generation): при вопросе он сначала ищет релевантные фрагменты закона в локальной базе (ChromaDB), затем передаёт их вместе с вопросом языковой модели (Ollama, работает полностью локально на твоём компьютере — никакие данные не уходят в облако). Каждый пользователь Telegram получает свой изолированный контекст диалога — переписка одного человека никогда не путается с перепиской другого, даже если они пишут одновременно из разных частей света.

## Стек

- Python 3.11
- ChromaDB — векторная база для поиска по тексту закона
- Ollama — локальная языковая модель (по умолчанию gemma3:4b)
- python-telegram-bot — интерфейс бота
- SQLite — логи и кеш ответов

## Запуск

1. Установи [Ollama](https://ollama.ai) и запусти его.
2. Скачай нужные модели:
   ```
   ollama pull gemma3:4b
   ollama pull nomic-embed-text
   ```
3. Скопируй `.env.example` в `.env` и впиши токен бота, полученный от `@BotFather` в Telegram.
4. Создай виртуальное окружение и установи зависимости:
   ```
   python -m venv .venv
   .venv\Scripts\activate
   pip install -r requirements.txt
   ```
5. Положи нормативно-правовые акты в формате `.docx` в папку `docs/` (УК РФ уже там).
6. Запусти бота:
   ```
   python telegram_bot.py
   ```

При первом запуске бот автоматически проиндексирует все `.docx` файлы из `docs/`. Чтобы добавить новый закон или кодекс — просто положи `.docx` файл в эту папку и перезапусти бота, он подхватит его сам.

## Команды бота

- `/start` — приветствие и краткая инструкция
- `/clear` — забыть текущий разговор и начать заново
- `/stats` — статистика использования (число запросов, уникальных пользователей, среднее время ответа)

## Логирование

Все взаимодействия с ботом записываются в файл `logs.db` (SQLite база, лежит в корне проекта рядом с `telegram_bot.py`). Это происходит автоматически — отдельно настраивать ничего не нужно, файл создаётся при первом запуске.

Что записывается в каждую строку: время запроса, идентификатор и имя пользователя Telegram, источник (telegram), сам вопрос, ответ бота, был ли ответ взят из кеша, и сколько миллисекунд занял ответ.

Посмотреть логи можно двумя способами:

1. Через Telegram командой `/stats` — покажет агрегированную статистику.
2. Открыв `logs.db` любой программой для SQLite, например [DB Browser for SQLite](https://sqlitebrowser.org/) (бесплатная, с понятным интерфейсом — установил, открыл файл, увидел таблицу).

Логи также можно выгрузить в `.csv` для Excel — это делается программно через метод `export_to_csv()` в `db_logger.py`.

## Кеш

Чтобы не гонять модель по одному и тому же простому вопросу заново, ответы кешируются в `cache.db` (тоже SQLite, отдельный от логов файл). Кеш используется только для вопросов без активного диалога — если ты уже общаешься с ботом и контекст важен, кеш автоматически пропускается, чтобы не выдать неправильный ответ из чужого контекста.

## Структура проекта

```
ADVOKAT/
├── telegram_bot.py     — точка входа, обработчики команд Telegram
├── main.py              — связующая логика (кеш + RAG + логи)
├── rag.py                — системный промпт, генерация ответов через Ollama
├── embeddings.py        — поиск по базе законов (ChromaDB)
├── cache.py              — кеш ответов
├── db_logger.py          — логирование в SQLite
├── config.py              — все настройки, читаются из .env
├── docs/                  — нормативно-правовые акты (.docx)
├── .env                   — токен и настройки (не в git)
├── .env.example          — шаблон .env
├── requirements.txt
└── test_basic.py          — тесты
```

## Известные ограничения

Это прототип, а не production-система. В частности отсутствуют: переранжирование результатов поиска (rerank), гибридный поиск (комбинация векторного и ключевого), сжатие длинного контекста, специализированный парсинг структуры закона (статья → часть → пункт) и отдельный контрольный слой проверки финального вывода модели. Для учебного и личного использования текущей точности поиска и качества модели достаточно; для масштабирования на реальных пользователей эти доработки стоит сделать в первую очередь.
